# 05 — Data & storage

lens keeps **relational state** in PostgreSQL and **large opaque content** (raw HTML and
unified diffs) in a blob store. Redis is used for ephemeral coordination state (locks,
throttle counters, idempotency keys, dead-letter lists) and is covered in
[Messaging & scaling](06-messaging-and-scaling.md).

## PostgreSQL schema

The ORM models live in `lens_infrastructure/db/models.py`. Repositories convert between
these rows and domain entities (`db/mapping.py`), so the rest of the system never sees
ORM objects.

| Table                    | Stores                                                                                                                             | Notes                                              |
|--------------------------|------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------|
| `domains`                | Host, display name, enabled flag, default crawl/diff/routing/politeness config                                                     | Aggregate root for defaults.                       |
| `categories`             | Per-domain grouping with optional config overrides                                                                                 |                                                    |
| `urls`                   | Address, schedule (`interval_seconds`, `next_due_at`), crawl state (`status`, lock fields, error counts), per-URL config overrides | Has a partial index on due URLs for the scheduler. |
| `snapshots`              | Per-fetch metadata: blob `content_ref`, content hash, HTTP status/size, `fetched_at`                                               | The HTML itself lives in the blob store.           |
| `changes`                | Detected diffs: line counts, `semantic_score`, `significant`, `enrichment_status`, `diff_ref`                                      | The diff text itself lives in the blob store.      |
| `channels`               | Notification channel name + kind; the Apprise URL stored **encrypted**                                                             | See secret handling below.                         |
| `channel_bindings`       | Channel → scope (global/domain/category/url) with trigger flags                                                                    | Drives notification routing.                       |
| `api_keys`               | Admin bearer tokens: name, **SHA-256 hash** (never the raw key), scopes, enabled                                                   | See [API reference](07-api-reference.md).          |
| `outbox`                 | Buffered domain events: aggregate metadata, unique `event_id`, JSON payload, `sent_at`, `attempts`                                 | The exactly-once publication buffer.               |
| `notification_log`       | Per-`(event_id, channel_id)` send record                                                                                           | Dedup so a notification is not sent twice.         |
| `site_profiles`          | Per-domain/URL-pattern template hash/class, zone selectors, significance rules, semantic threshold, version                        | Powers pipeline L2–L4.                             |
| `url_check_states`       | Prior-check pipeline state per URL: raw hash, etag/last-modified, per-zone hashes/texts, previous cleaned text                     | Lets L0/L1/L3 short-circuit.                       |
| `change_classifications` | One LLM classification per change                                                                                                  | AI tier.                                           |
| `change_labels`          | Human/LLM/rule labels per change, for learning & evaluation                                                                        | AI tier.                                           |
| `zone_embeddings`        | Cached embedding vectors keyed by `(model_id, text_hash)`                                                                          | AI tier embedding cache.                           |

### Identity and time conventions

- All primary keys are **UUIDv7** (time-ordered UUIDs), generated by the application.
- All timestamps are **timezone-aware UTC**.

## Migrations

Schema is managed with **Alembic**. The configuration lives at
`libs/infrastructure/migrations/alembic.ini` and the revision chain runs from
`0001_initial` through the pipeline, notifications, site-profile, AI-enrichment, and
auto-learning revisions.

Run migrations with either:

```bash
make migrate                 # alembic upgrade head against libs/infrastructure/migrations
# or
LENS_DATABASE_URL=... uv run lens migrate
```

The Alembic environment reads the database URL from the environment, and the CLI
`migrate` command injects it from `LENS_DATABASE_URL`. See
[Configuration](09-configuration.md).

> Maintainer note: a small number of ORM-defined fields/tables introduced late
> (`url_check_states`, `changes.enrichment_status`) should be verified against the
> migration head when bringing up a fresh database; create the missing revision if the
> ORM metadata and the migrated schema diverge.

## Blob storage

Raw HTML snapshots and unified diffs are stored as **gzip-compressed blobs** behind the
`BlobStoragePort`. The shipped adapter (`LocalFileBlobStorage` / its async wrapper)
writes to the local filesystem under a configurable root, using the blob key as a
relative path (e.g. `diffs/{change_id}.diff.gz`).

A bucket/object-store backend (S3/MinIO) is anticipated by the deployment configuration
and dependencies; the local filesystem adapter is what the code wires today. For
multi-instance deployments, point the blob root at a shared volume or substitute an
object-store adapter behind the same port.

## Retention

Storage is bounded by **per-URL snapshot retention**: the `EnforceRetentionUseCase`
keeps at most `MAX_SNAPSHOTS` snapshots per URL and deletes the excess rows *and* their
backing blobs. A companion `SweepOrphanBlobsUseCase` deletes blob keys that are no
longer referenced by any snapshot. Both are exposed as admin endpoints (see
[API reference](07-api-reference.md)) and can be scheduled operationally.

## Pagination

List operations use **cursor pagination** (`PageRequest`/`Page`), with the last seen id
as the cursor. This keeps API responses bounded and stable across inserts.

## What lives where (summary)

| Concern | Store |
| --- | --- |
| Domains, categories, URLs, channels, bindings, changes, snapshots (metadata), api keys, outbox, notification log, site profiles, check state, classifications, labels | PostgreSQL |
| Raw HTML, unified diff text | Blob store (local filesystem by default) |
| Distributed locks, per-host throttle, idempotency keys, dead-letter lists | Redis |
| Crawl tasks, domain events | RabbitMQ (in flight only) |
